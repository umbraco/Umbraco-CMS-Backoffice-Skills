# CLAUDE.md

Guidance for working in this repository.

## What this repo is

A **Claude Code plugin marketplace** of Umbraco backoffice skills — documentation plus **validated, working code examples** (TypeScript/Lit manifests and C#) for building Umbraco backoffice extensions. It is not an application; the "product" is the SKILL.md files and their examples.

Two plugins (see `.claude-plugin/marketplace.json`):
- **`umbraco-cms-backoffice-skills`** (`plugins/umbraco-backoffice-skills/`, ~65 skills) — Foundation concepts, Extension Types, Property Editors, Rich Text, Backend, Setup.
- **`umbraco-cms-backoffice-testing-skills`** (`plugins/umbraco-testing-skills/`, ~8 skills) — unit, MSW/mocked, and E2E testing.

Both are versioned to the targeted Umbraco major. This is the **v17 line** (`main` targets v18); both plugins are at `17.5.0`, and the host references `Umbraco.Cms 17.*` (currently resolving to 17.5.x). When bumping, keep `marketplace.json` and both `plugin.json` versions in sync.

## Layout

- `plugins/<plugin>/skills/<skill-name>/SKILL.md` — one skill each; some have an `examples/` subfolder with a full working extension (C# project + `Client/` TypeScript).
- `Umbraco-CMS.Skills/` — a real Umbraco web host (`net10.0`) that references the four backoffice examples (Blueprint, TimeDashboard, tree-example/UmbTreeClient, notes-wiki) so they compile and run. SQLite, unattended install; admin `admin@example.com` / `1234567890`; serves `https://localhost:44325`.
- `.claude/skills/` — the repo's own validation tooling (`umbraco-skill-validator`, `umbraco-skill-code-analyzer`, `umbraco-skill-test-runner`).
- Root `*-report.json` files are generated by the validators and are **gitignored** — never commit them.

## Build

```bash
dotnet build Umbraco-CMS.Skills/Umbraco-CMS.Skills.csproj -c Debug
```
This restores v17 NuGet, compiles all example C# projects, and builds each example's `Client/` via vite (output committed under each example's `wwwroot/App_Plugins/`). Targets `net10.0`. The `Clean` starter-kit package must match the Umbraco major (v17 → `Clean 7.0.1`).

## Run the instance

```bash
ASPNETCORE_ENVIRONMENT=Development dotnet run --project Umbraco-CMS.Skills/Umbraco-CMS.Skills.csproj -c Debug
```
Backoffice at `https://localhost:44325/umbraco`. The DB auto-upgrades on boot. (The public front-end homepage may 500 via InMemoryAuto Razor compilation — unrelated to the backoffice, which is what these skills cover.)

## Worktrees

Creating a Claude Code worktree fires hooks (`.claude/settings.json` → `scripts/worktree-create.sh` / `worktree-remove.sh`) that let a worktree's Umbraco host run **alongside** the primary checkout without a port clash:

- **Dynamic host port.** The create hook rewrites the worktree's `Umbraco-CMS.Skills/Properties/launchSettings.json` to port `0`, so the OS assigns a free ephemeral port instead of the fixed `44325`/`60290`. (Local-only edit inside the worktree — don't commit it.) The test-runner discovers the actual URL from the host's `Now listening on: …` boot log; it never reuses another checkout's host when the port is dynamic. If you start the host manually in a worktree, read the port it prints.
- **DB is already isolated.** The host uses SQLite with a gitignored DB file, so each worktree does its own unattended install into its own DB.
- **Mocked dev-server ports** (`5174`/`5175`/`5176`) are overridable via the `DEV_SERVER_PORT` env var, so concurrent worktrees running mocked suites don't collide.
- Worktrees live under `.claude/worktrees/<slug>` (gitignored). The remove hook kills any process holding the worktree (host, vite, Playwright) before `git worktree remove`.

## Validation (run before considering skill changes done)

```bash
# links + skill/path references
cd .claude/skills/umbraco-skill-validator/scripts && npm install --silent && npx tsx validate-links.ts
# code examples: imports, extension types, deprecated patterns, TS compile
cd .claude/skills/umbraco-skill-code-analyzer/scripts && npm install --silent && npx tsx analyze-code.ts   # CHECK_TYPESCRIPT=false to skip tsc
# ALL test suites — unit + mocked + E2E. E2E runs against the real host below.
cd .claude/skills/umbraco-skill-test-runner/scripts && npm install --silent && \
URL=https://localhost:44325 UMBRACO_USER_LOGIN=admin@example.com UMBRACO_USER_PASSWORD=1234567890 \
UMBRACO_CLIENT_PATH=/path/to/Umbraco-CMS/src/Umbraco.Web.UI.Client \
npx tsx run-tests.ts
```

Prefer the `/validate-skills` command, which runs all three phases. **Full validation
includes running the Umbraco instance and ALL E2E tests** — it is not a pass if E2E is
`skipped`. The test runner **auto-starts** the `Umbraco-CMS.Skills` host (`dotnet run`,
`https://localhost:44325`) when it isn't already up and stops it afterward, but it only
runs E2E when the credentials above are supplied (repo defaults: `admin@example.com` /
`1234567890`). The mocked-backoffice suites additionally need `UMBRACO_CLIENT_PATH`
pointing at a built v17 client (e.g. a `release/17.x` checkout of `Umbraco-CMS`, `npm ci`
in `Umbraco.Web.UI.Client`). If you'd rather run the host yourself, start it with the
`dotnet run` command in "Run the instance" above, then run the suite.

Notes:
- The analyzer is a **heuristic checker** with hardcoded `KNOWN_BACKOFFICE_MODULES` / `KNOWN_EXTENSION_TYPES` / `DEPRECATED_PATTERNS` in `analyze-code.ts` — update these lists when the backoffice API changes (it does not type-check against the real client).
- The validator's 3 `missing-skill` warnings for `umbraco-extension-reviewer` are **false positives** — it's an agent (`plugins/umbraco-backoffice-skills/agents/`), not a skill.

## Testing model

Example test suites (under each example's `Client/tests/` or the testing-skill `examples/`):
- **Unit** — `@open-wc/testing` via web-test-runner (`npm test`). No instance needed.
- **Mocked backoffice** — Playwright against the **Umbraco client dev server** (MSW). Requires `UMBRACO_CLIENT_PATH` pointing at a built `Umbraco.Web.UI.Client`, launched via `VITE_EXAMPLE_PATH=<abs> VITE_UMBRACO_USE_MSW=on npm run dev`. A self-contained harness (`plugins/umbraco-testing-skills/harness/mocked-backoffice/`, wired as Playwright `globalSetup`/`globalTeardown`) **patches a stock client and reverts afterward** — so a stock v17 client works, no CMS branch required. It injects **both** halves of the (unmerged upstream) `external-extension-dev-support` work: (1) external-extension loading (absolute `VITE_EXAMPLE_PATH` via `/@fs/` + shared-package resolver in `index.ts`/`vite.config.ts`), and (2) `window.MockServiceWorker.addMockHandlers(...)` in `mocks/index.ts`, which an example needs to mock its **own** custom API endpoint (e.g. a custom tree). Omitting (2) makes such an example load but silently render nothing. First cold run may re-optimize deps (flaky first navigation); configs set `retries`.
- **E2E** — Playwright against the **real running instance** at `https://localhost:44325`. Run with `URL=https://localhost:44325 UMBRACO_USER_LOGIN=admin@example.com UMBRACO_USER_PASSWORD=1234567890 npx playwright test`.

Testhelpers package: **`@umbraco-cms/acceptance-test-helpers`** (pinned `^17.5.1`). This is the rename of the old `@umbraco/playwright-testhelpers`: from Umbraco 17.3 the helpers moved into the CMS acceptance-test project and are published under the new name, versioned to match CMS releases. Use the renamed package.

### Discovery contract (keep this and `discoverTestableExamples()` in sync)

The runner globs `**/examples/**/package.json` and maps script names to suite types:
- `test` containing `web-test-runner` → **unit**
- `test:mocked` / `test:msw` / `test:mock-repo` → **mocked**
- `test:e2e`, or `test` containing `playwright` (and not web-test-runner) → **e2e**

**Adding a new suite** = add a script whose name matches the above (or teach `discoverTestableExamples()` a new name). Interactive variants (`:watch` / `:headed` / `:ui` / `:debug` / `:report`) are intentionally ignored. A **coverage guard** (`auditTestCoverage()`) fails the run (exit 2, `orphans` in `test-report.json`) if any `test*` script that invokes playwright/web-test-runner, or any `*.spec.ts`/`*.test.ts` under `examples/`, isn't covered by a discovered suite — so a silently-skipped suite (as `test:msw`/`test:mock-repo` once were) can't recur.

## Umbraco v17 gotchas (learned the hard way)

- **Auth is httpOnly-cookie based** (as of 17.5). The access token is no longer in `localStorage`; the Management API needs a Bearer token that only the SPA holds. Don't hand-roll cookie API calls (they 401). Use `@umbraco-cms/acceptance-test-helpers` (its `isLoginStateValid` refreshes via the token endpoint).
- **Tree data-source paging + return shape.** `args.skip`/`args.take` are **deprecated since 16.3** (use `args.paging`, offset- or target-based) but still work in v17. The data source's `getRootItems`/`getChildrenOf` must return a `UmbTargetPagedModel` — i.e. include `totalBefore`/`totalAfter` (pad to `0` for flat trees). The tree examples use `UmbTreeServerDataSourceBase` and normalise both via `toOffsetQuery(args.paging)` + a `toTargetPaged()` wrapper.
- **Backoffice API client** only gets the auth token if a `backofficeEntryPoint` manifest is registered in the bundle the real host loads (`bundle.manifests.ts`), not just the external `index.ts`. Without it every Management-API call from the extension 401s against a real host.
- **Property editor value events**: dispatch `UmbChangeEvent` from `@umbraco-cms/backoffice/event`, not the legacy `property-value-change` CustomEvent.
- **OpenAPI (v17)** still uses **Swashbuckle** — register a custom backoffice API document via `SwaggerGenOptions` / `UmbracoPipelineOptions` (see `umbraco-openapi-client`). (This is the pattern that changes in v18; keep the Swashbuckle pattern for v17.)
- **Docs URLs** for v17 use a version segment: `docs.umbraco.com/umbraco-cms/17.latest/extend-your-project/backoffice-extensions/...`.

## Conventions

- Match the surrounding skill's structure and tone; SKILL.md files link to official docs and to other skills by name (e.g. `` `umbraco-dashboard` ``).
- Commit generated example client assets (`wwwroot/App_Plugins/`) when their source changes; do not commit report JSONs, `node_modules`, or Playwright `test-results`/`.auth`.
